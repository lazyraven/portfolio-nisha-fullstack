Complete Learning Path

AWS DevOps Roadmap

Interview-ready · Real-world focused · 42-day plan

Step 01
🏗️ Core DevOps Concepts
DevOps Lifecycle
  • Plan → Code → Build → Test → Deploy → Monitor
  • Teams collaborate via Git + automated pipelines
  • Automation reduces manual deployment errors
  • Version control: Feature branch, trunk-based
  • Linux basics: permissions, processes, networking
Q: Why is DevOps important?
A: Reduces release time, improves reliability, and enables continuous delivery through automation.
CI/CD Pipelines
  • Pipeline stages: Build → Test → Package → Deploy
  • CI vs CD vs Continuous Deployment
  • Tools: Jenkins, GitHub Actions
  • Pipeline as Code (YAML format)
  • Rollbacks & failure handling strategies
Q: How do you handle a failed deployment?
A: Rollback to previous stable version, use health checks, set up automated alerts.
Git Workflows
  • Feature branch workflow — isolated development
  • Trunk-based development — fast integration
  • Pull requests + code reviews
  • Commit conventions & tagging
  • Merge strategies: squash, rebase, merge
Step 02 — HIGH PRIORITY
☁️ AWS Core Services
Compute
  • EC2 — Virtual machines, scalable compute
  • Lambda — Serverless, event-driven functions
  • ECS / EKS — Managed containers & Kubernetes
  • Auto Scaling Groups for high availability
  • Spot instances for cost optimization
Q: EC2 vs Lambda?
A: EC2 for persistent, long-running workloads. Lambda for short, event-driven, stateless functions with auto-scaling.
Storage & Database
  • S3 — Object storage, static hosting, artifacts
  • EBS — Block storage attached to EC2
  • RDS — Managed SQL (MySQL, PostgreSQL)
  • DynamoDB — Serverless NoSQL
  • S3 storage classes: Standard, IA, Glacier
Networking
  • VPC — Isolated virtual network
  • Subnets: public vs private
  • ALB / NLB — Application & Network Load Balancer
  • Route Tables + Internet Gateway
  • Security Groups vs Network ACLs
Real-World Combos
  • EC2 + ALB + RDS — Traditional web app
  • Lambda + API Gateway + DynamoDB — Serverless API
  • ECS + ECR + ALB — Containerized app
  • S3 + CloudFront — Static site CDN
  • CodePipeline + CodeBuild + ECS — Full CI/CD
Step 03
⚙️ Infrastructure as Code + Containers
Terraform
  • Write declarative HCL configs
  • Create EC2, VPC, subnets via code
  • Variables, modules, outputs
  • Remote state in S3 + DynamoDB lock
  • Plan → Apply → Destroy workflow
Q: Why IaC?
A: Repeatability, version control, automated provisioning, and reduced human error.
CloudFormation
  • AWS-native IaC using YAML/JSON templates
  • Stacks, nested stacks, change sets
  • Rollback on deployment failure
  • Integrates natively with all AWS services
  • CDK — define infra using Python/TypeScript
Docker
  • Dockerfile: FROM, RUN, COPY, EXPOSE, CMD
  • Build image → Push to ECR registry
  • Multi-stage builds for smaller images
  • docker-compose for local dev
  • Layer caching best practices
ECS / EKS
  • ECS Fargate — serverless containers
  • Task Definitions & Services
  • EKS — managed Kubernetes on AWS
  • Pods, Deployments, Services, Ingress
  • Horizontal Pod Autoscaler
Step 04
📊 Monitoring, Logging & Security
Monitoring Tools
  • CloudWatch — Metrics, logs, alarms, dashboards
  • Prometheus + Grafana — Open-source monitoring
  • ELK Stack — Elasticsearch, Logstash, Kibana
  • CPU, memory, request latency alerts
  • Centralized log aggregation
IAM & Security
  • IAM — Users, groups, roles, policies
  • Least privilege access principle
  • Never hardcode secrets in code
  • Secrets Manager / SSM — Secure secrets storage
  • Security Groups vs NACLs
Q: IAM best practices?
A: Least privilege, use roles not users for services, rotate keys, enable MFA, use Secrets Manager.
Deployment Strategies
  • Blue-Green — Two environments, instant swap
  • Canary — Gradual traffic shift (5% → 100%)
  • Rolling — Replace instances incrementally
  • Feature flags for controlled releases
  • Health check gates before full rollout
