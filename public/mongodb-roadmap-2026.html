Fundamentals

Core Concepts
NoSQL vs SQL
  • ▸NoSQL = Not Only SQL — flexible data
  • ▸Document-based model — JSON/BSON format
  • ▸Collections ≈ Tables · Documents ≈ Rows
  • ▸Dynamic schema — no rigid column definitions
  • ▸Horizontal scaling by design
SQL vs MongoDB mental model Table → Collection · Row → Document · Column → Field · JOIN → $lookup
Basic Operations ⭐
CRUD — Create, Read, Update, Delete
  • ▸insertOne() — single document
  • ▸insertMany() — bulk insert
  • ▸find() / findOne() — read
  • ▸updateOne() / updateMany()
  • ▸deleteOne() / deleteMany()
Query Operators
Filters & Projections
  • ▸Comparison: $eq, $gt, $lt, $gte, $ne
  • ▸Logical: $and, $or, $not, $nor
  • ▸Array: $in, $nin, $all
  • ▸Projection — select specific fields
  • ▸Sorting, limit, skip for pagination
crud-basics.js — MongoDB CRUD operations
MongoDBNode.js
// INSERT
await db.collection('students').insertOne({
  name: "Aarav", grade: 10, score: 92
});

// READ with query operators
const topStudents = await db.collection('students')
  .find({ score: { $gte: 85 } })
  .sort({ score: -1 })
  .limit(5)
  .toArray();

// UPDATE — only the matched fields
await db.collection('students').updateOne(
  { name: "Aarav" },
  { $set: { score: 95 } }
);

// DELETE
await db.collection('students').deleteOne({ score: { $lt: 40 } });

Intermediate
// Indexing + Schema + Aggregation
Performance 🔥
Indexing
  • ▸Single field index — fastest lookup
  • ▸Compound index — multiple fields together
  • ▸Unique index — enforce uniqueness
  • ▸Text index — full-text search
  • ▸TTL index — auto-expire documents
Interview: "How does MongoDB optimize queries?" Indexes allow O(log n) lookup instead of full collection scan O(n).
Real-World Critical 🔥
Schema Design
  • ▸Embedded — store related data together
  • ▸Referenced — store IDs, join via $lookup
  • ▸One-to-One: always embed
  • ▸One-to-Many: embed if small, reference if large
  • ▸Many-to-Many: reference with junction or arrays
Interview: "Embed vs Reference?" Embed when data is always read together. Reference when data is large, shared, or updated often.
Data Patterns
Schema Design Patterns
  • ▸Bucket — group time-series data in arrays
  • ▸Subset — store only frequently accessed fields
  • ▸Outlier — handle edge-case large documents
  • ▸Computed — pre-calculate expensive values
  • ▸Polymorphic — different shapes, same collection
✅ Embed — Use When
  • ▸Data always accessed together
  • ▸Small, bounded sub-documents
  • ▸Data doesn't change frequently
  • ▸No need to query sub-document alone
  • ▸Example: User + address
🔗 Reference — Use When
  • ▸Data is shared across documents
  • ▸Sub-document grows unbounded
  • ▸Need to query child docs independently
  • ▸Frequent updates to related data
  • ▸Example: Post + comments (many)
agg
🔥 Must Master — VERY Important
Aggregation Framework
$match
Filter documents like a WHERE clause
$group
Group by field, compute sums/counts
$project
Reshape output, include/exclude fields
$lookup
Join documents from another collection
$unwind
Deconstruct array into separate docs
$sort
Sort by any field ascending/descending
$limit
Limit output size
aggregation.js — Total sales per user
AggregationMongoDB
// Find top 3 users by total sales
const result = await db.collection('orders').aggregate([
  { $match: { status: "completed" } },    // filter
  { $group: {
      _id: "$userId",
      totalSales: { $sum: "$amount" },     // aggregate
      orderCount: { $sum: 1 }
  }},
  { $sort: { totalSales: -1 } },          // sort desc
  { $limit: 3 },                           // top 3
  { $lookup: {                               // join users
      from: "users",
      localField: "_id",
      foreignField: "_id",
      as: "userInfo"
  }}
]).toArray();

🟠 Stand Out Level
Advanced Topics
// Replicas · Sharding · ACID
Query Analysis
Performance Optimization
  • ▸explain() — see query execution plan
  • ▸Check: COLLSCAN (bad) vs IXSCAN (good)
  • ▸Index selectivity — high = more efficient
  • ▸Covered queries — answered entirely by index
  • ▸Avoid: $where, regex without prefix
ACID Compliance
Transactions
  • ▸Multi-document transactions (v4.0+)
  • ▸session.startTransaction()
  • ▸commitTransaction / abortTransaction
  • ▸Use for: financial ops, cart + inventory
  • ▸Performance cost — use sparingly
High Availability
Replication
  • ▸Replica set — group of MongoDB instances
  • ▸Primary — handles all writes
  • ▸Secondary — replicates from primary
  • ▸Automatic failover — secondary becomes primary
  • ▸Reads from secondaries — reduce primary load
Horizontal Scaling
Sharding
  • ▸Split data across multiple servers (shards)
  • ▸Shard key — determines data distribution
  • ▸Chunk distribution — data in range chunks
  • ▸mongos — query router for sharded cluster
  • ▸Good shard key = even distribution
Access Control
Security
  • ▸Authentication — SCRAM, x.509
  • ▸Authorization — role-based access control
  • ▸Built-in roles: readWrite, dbAdmin, root
  • ▸Encryption at rest + in transit (TLS)
  • ▸Field-level encryption (sensitive data)

user.model.js — Mongoose schema example
MongooseSchema
const mongoose = require('mongoose');

const userSchema = new mongoose.Schema({
  name:  { type: String, required: true, trim: true },
  email: { type: String, required: true, unique: true, lowercase: true },
  age:   { type: Number, min: 0, max: 120 },
  posts: [{ type: mongoose.Schema.Types.ObjectId, ref: 'Post' }]
}, { timestamps: true });

// Pre-save middleware — hash password
userSchema.pre('save', async function() {
  if (this.isModified('password')) {
    this.password = await bcrypt.hash(this.password, 12);
  }
});

// Virtual — fullName computed field
userSchema.virtual('postCount').get(function() {
  return this.posts.length;
});

module.exports = mongoose.model('User', userSchema);

Interview-Focused Topics
CRUD operations Aggregation pipeline Indexing Schema design SQL vs MongoDB Mongoose populate Sharding Replica sets Embedded vs Referenced Query performance $lookup (joins) Pagination
1
How does MongoDB store data internally?
BSON (Binary JSON) format — supports more data types than JSON. Stored in collection files on disk. Documents up to 16MB max.
2
Difference between embedded and referenced documents?
Embedded = store sub-data inside parent (one read). Referenced = store IDs, use $lookup (two reads). Choose based on read patterns and data size.
3
What is an aggregation pipeline?
A chain of stages that process documents in sequence. Each stage transforms data. Output of one stage = input of next. Powerful for analytics and reporting.
4
How do indexes improve performance?
Indexes create a sorted data structure (B-tree) that allows O(log n) lookup instead of O(n) collection scan. Dramatically faster for large collections.
5
What is sharding in MongoDB?
Horizontal scaling — split collection data across multiple servers (shards) using a shard key. Enables handling datasets larger than a single server.
6
Explain replica sets.
A group of MongoDB instances maintaining the same data. One primary (writes), multiple secondaries (replicate from primary). Automatic failover if primary goes down.
7
Write a query to find top 3 users by activity.
Use aggregation: $group by userId with $sum, $sort descending, $limit 3, $lookup to join user details.

Common Gaps
What Most People Miss
❌ Mistake #1
Skipping Aggregation
  • ▸Aggregation is asked in almost every interview
  • ▸Real apps need it for analytics, reports
  • ▸Without it, you can't do MongoDB "joins"
  • ▸Practice at least 10 pipeline problems
❌ Mistake #2
Ignoring Schema Design
  • ▸Bad schema = slow queries forever
  • ▸Design for how you read, not how you write
  • ▸Most bugs come from wrong data modeling
  • ▸Practice embed vs reference decisions daily
❌ Mistake #3
Theory Without Projects
  • ▸Watching videos ≠ understanding MongoDB
  • ▸Build real APIs — experience the pain points
  • ▸Every project teaches a lesson videos can't
  • ▸Focus on: hands-on + real queries always
